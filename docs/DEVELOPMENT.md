# Development

This repository is the unified content hub for skills, plugins, assistants, and MCP servers. Root npm scripts are the supported entry points for creating content, refreshing category registries, and validating the hub.

## Scripts

```sh
npm run new:skill -- contract-helper
npm run new:plugin -- data-tools
npm run new:assistant -- research-helper
npm run new:mcp -- local-search
npm run sync
npm run validate
```

The scaffold commands create content from the matching category template under `<category>/templates/`, refresh the generated registries, and print the created path. Content IDs must match `^[a-z][a-z0-9-_]*$`.

## Registries

Every category is registered in a marketplace manifest using the `.agents` marketplace format (the same shape as the existing plugins registry):

- `skills/.agents/skills/marketplace.json` — generated by `npm run sync`
- `plugins/.agents/plugins/marketplace.json` — **authoritative, hand-maintained** (the plugin scaffold appends to it; `npm run sync` only checks it against the plugins on disk)
- `assistants/.agents/assistants/marketplace.json` — generated by `npm run sync`
- `mcp/.agents/mcp/marketplace.json` — generated by `npm run sync`

Each entry follows `{ "name": "<id>", "source": { "source": "local", "path": "<relative path>" } }`; plugin entries additionally carry `policy` and `category`. Registry paths are relative to the file's own category root.

Per-category `version` values live in the top-level `index.json` and are maintained manually.

## Templates

One template directory per category, named uniformly:

- `skills/templates/basic-skill/`
- `plugins/templates/basic-plugin/`
- `assistants/templates/basic-assistant/`
- `mcp/templates/basic-mcp/`

`templates/`, `.agents/`, and `_`-prefixed directories are never treated as content.

## Adding Content

1. Create the scaffold with the relevant `npm run new:* -- <id>` command.
2. Edit the generated content files.
3. Run `npm run sync` to refresh the generated registries.
4. Manually bump the affected category `version` in the top-level `index.json` when the change should be published.
5. Run `npm run validate`.
6. Commit the scaffold, edited files, generated registry updates, and any manual version bump together.

## Local MCP package release

MCP adapter source is committed to Git. Generated `.tgz` files stay under the ignored `dist/` directory and are uploaded as GitHub Release assets; do not commit generated packages to the repository.

The complete local pipeline validates the hub, audits runtime dependencies, bundle-checks all 1,075 adapters, builds the selected packages, verifies their SHA-256 hashes, and uploads resumable immutable releases:

```powershell
$env:GITHUB_TOKEN = "<fine-grained token with Contents: read and write>"
npm ci
npm run release:mcp:local -- --selection all --version 1.0.0 --concurrency 8
Remove-Item Env:GITHUB_TOKEN
```

Each adapter is published under `mcp-<service>-v<version>` with its service-only package name, for example `aliyun-oss-1.0.0.tgz`. An already complete release is skipped when resuming. A published but incomplete release is never overwritten; use a new version after resolving it.

To build without uploading, run:

```powershell
npm run build:mcp:all -- --version 1.0.0 --concurrency 8
```
